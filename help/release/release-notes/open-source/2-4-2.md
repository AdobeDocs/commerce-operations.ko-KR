# Magento Open Source 2.4.2 릴리스 노트

Magento Open Source 2.4.2에서는 성능 및 보안 개선 사항과 상당한 플랫폼 개선 사항이 도입되었습니다. 보안 개선 사항에는 `SameSite` 모든 쿠키에 대한 속성입니다. 이제 Elasticsearch 7.9.x 및 Redis 6.x가 지원됩니다.

이 릴리스에는 코어 코드와 35개의 보안 개선 사항에 대한 280개 이상의 새로운 수정 사항이 포함되어 있습니다. 여기에는 커뮤니티 구성원의 약 290개의 GitHub 문제 해결도 포함됩니다. 이러한 커뮤니티 기여는 사소한 코어 코드 정리부터 GraphQL의 중요한 개선 사항까지 다양합니다.

2.4.1에서 식별된 모든 알려진 문제는 이 릴리스에서 해결되었습니다.

{{bics}}

## 적용 `AC-3022.patch` DHL을 계속 운송 운송업체로 제공

DHL에서 스키마 버전 6.2를 도입했으며 조만간 스키마 버전 6.0을 사용하지 않을 예정입니다. DHL 통합을 지원하는 Adobe Commerce 2.4.4 및 이전 버전은 버전 6.0만 지원합니다. 이러한 릴리스를 배포하는 판매업체는 이 버전을 적용해야 합니다 `AC-3022.patch` 그들이 편의상 DHL을 운송 회사로 계속 제공할 수 있도록. 자세한 내용은 [DHL을 배송 반송파로 계속 제공하기 위해 패치를 적용합니다.](https://support.magento.com/hc/en-us/articles/7707818131597-Apply-a-patch-to-continue-offering-DHL-as-shipping-carrier) 패치 다운로드 및 설치에 대한 자세한 내용은 기술 자료 문서를 참조하십시오.

## 보안 패치 사용 가능

이제 판매자는 전체 분기별 릴리스에서 제공하는 수백 개의 기능 수정 사항 및 개선 사항을 적용하지 않고 시간에 민감한 보안 수정 사항을 설치할 수 있습니다(예: 2.4.1-p1). 패치 2.4.0.12(작성기 패키지 2.4.1-p1)는 이전 분기별 릴리스 2.4.1에서 식별된 취약점에 대한 수정 사항을 제공하는 보안 패치입니다. 2.4.1 릴리스에 적용된 모든 핫픽스는 이 보안 패치에 포함됩니다. (A) *핫픽스* 은 특정 문제 또는 버그를 해결하는 릴리스된 버전에 대한 수정 사항을 제공합니다.)

보안 패치에 대한 일반적인 정보는 다음을 참조하십시오 [새 보안 패치 릴리스 소개](https://community.magento.com/t5/Magento-DevBlog/Introducing-the-New-Security-Patch-Release/ba-p/141287). 보안 패치(패치 2.4.1-p1 포함)를 다운로드하고 적용하는 방법에 대한 지침은 [온-프레미스 설치 빠른 시작](../../../installation/composer.md). 보안 패치에는 보안 버그 수정만 포함되며, 전체 패치에 포함된 추가 보안 개선 사항은 포함되지 않습니다.

## 기타 릴리스 정보

이러한 기능에 대한 코드가 분기별 릴리스와 함께 번들로 제공되지만, 이러한 프로젝트(예: Progressive Web Application(PWA) Studio)의 일부 도 독립적으로 릴리스됩니다. 이러한 프로젝트에 대한 버그 수정 사항은 각 프로젝트의 설명서에서 사용할 수 있는 프로젝트별 릴리스 정보에 설명되어 있습니다.

## 밝은 영역

이 릴리스에서 다음 주요 내용을 살펴보십시오.

### 대폭 향상된 보안 기능

이 릴리스에는 35개가 넘는 보안 수정 사항과 플랫폼 보안 개선 사항이 포함되어 있습니다. 모든 보안 수정 사항이 2.4.1-p1 및 2.3.6-p1로 지원되었습니다.

#### RCE(원격 코드 실행) 및 XSS(크로스 사이트 스크립팅) 취약점을 해결하는 데 도움이 되는 35개 이상의 보안 개선 사항

지금까지 이 문제와 관련된 확인된 공격은 발생하지 않았습니다. 그러나 특정 취약점은 고객 정보에 액세스하거나 관리자 세션을 인계하기 위해 악용될 수 있습니다. 이러한 문제 대부분은 공격자가 먼저 관리자에 액세스할 수 있어야 합니다. 그 결과, 다음 작업을 포함하되 이에 제한되지 않는 등 관리자를 보호하기 위해 필요한 모든 조치를 취하도록 요구합니다. IP 허용 목록에 추가, [이중 인증](https://devdocs.magento.com/guides/v2.4/security/two-factor-authentication.html), VPN 사용, 고유한 위치 사용 `/admin`그리고 암호 위생도 좋습니다 자세한 내용은 [Adobe 보안 공지](https://helpx.adobe.com/security/products/magento/apsb21-08.html) 이 해결된 문제에 대해 토론해 보십시오.

#### 추가적인 보안 개선 사항

이 릴리스의 보안 개선 사항은 다음과 같습니다.

* 이제 모든 코어 쿠키가 `SameSite` 속성을 사용합니다.

* 이제 사용자가 이러한 필드에 값을 저장하려고 하면 응용 프로그램에서 제품 및 카테고리 설명 필드에 악의적인 콘텐츠를 식별하는 메시지가 표시됩니다.

* 구성 요소 간의 파일 시스템 작업은 악성 업로드를 방지하기 위해 표준화되고 제어되었습니다.

* 핵심 CSP(콘텐츠 보안 정책) 위반이 수정되었습니다.

### 인프라 개선

이 릴리스에는 Framework 및 이러한 기능 영역의 품질을 향상시키는 코어 품질 개선 사항이 포함되어 있습니다. 고객 계정, 카탈로그, CMS, OMS, 가져오기/내보내기, 프로모션 및 타겟팅, 장바구니 및 체크아웃.

### 플랫폼 개선 사항

* [**이제 Elasticsearch 7.9.x가 지원됩니다**](../../../installation/system-requirements.md). Elasticsearch 7.9.x를 실행하는 것이 좋지만 버전 2.4.x는 Elasticsearch 7.4.x와 호환됩니다. <!-- MC-36867-->

* 2.4.2는 [Varnish 6.4](../../../installation/system-requirements.md). 버전 2.4.x는 Varnish 6.x와 호환됩니다.

* [**이제 Redis 6.x가 지원됩니다**](../../../installation/system-requirements.md). 버전 2.4.x는 Redis 5.x와 호환됩니다. <!-- MC-34853-->

* 2.4.2는 이제 와 호환됩니다. **Composer 2.x**. 판매자가 Composer 2.x로 마이그레이션하는 것이 좋습니다. Composer 1.x를 사용하여 이 릴리스를 설치할 수 있지만 Composer 1.x는 곧 사용이 종료됩니다. 작성기 2.x 기능에 대한 개요는 다음을 참조하십시오 [이제 작성기 2.0을 사용할 수 있습니다.](https://blog.packagist.com/composer-2-0-is-now-available/)

분할 데이터베이스를 사용하도록 설치를 구성하는 기능은 이 릴리스에서 더 이상 사용되지 않습니다. 현재 분할 데이터베이스를 사용하는 판매자는 단일 데이터베이스로 되돌리기 또는 마이그레이션하거나 대체 방법을 사용할 계획을 세워야 합니다. 자세한 내용은 [Magento Open Source에서 데이터베이스 분할 기능 사용 중단](https://community.magento.com/t5/Magento-DevBlog/Deprecation-of-Split-Database-in-Magento-Commerce/ba-p/465187) 이 문제에 대한 개요에 대한 DevBlog 게시물입니다.  자세한 내용은 [분할 데이터베이스에서 단일 데이터베이스로 되돌리기](../../../configuration/storage/revert-split-database.md) 마이그레이션 지침

### 향상된 성능

이 릴리스에는 큰 카탈로그를 사용하는 배포에 대한 API 성능 및 관리자 응답 시간을 높이는 코드 개선 사항이 포함되어 있습니다. 여러 가지 확장성 개선 사항을 통해 2.4.2는 이전 릴리스보다 최대 20배 더 큰 복잡한 카탈로그를 기본적으로 지원할 수 있습니다.

### GraphQL

이 릴리스는 다음 기능에 대한 GraphQL 범위를 추가합니다.

* 에 대한 지원이 추가되었습니다 [비교 목록](https://devdocs.magento.com/guides/v2.4/graphql/queries/compare-list.html). 쇼핑객은 다음을 수행할 수 있습니다 [만들기](https://devdocs.magento.com/guides/v2.4/graphql/mutations/create-compare-list.html) 및 [delete](https://devdocs.magento.com/guides/v2.4/graphql/mutations/delete-compare-list.html) 비교 목록 및 [추가](https://devdocs.magento.com/guides/v2.4/graphql/mutations/add-products-to-compare-list.html) 및 [제거](https://devdocs.magento.com/guides/v2.4/graphql/mutations/remove-products-from-compare-list.html) 항목을 비교 목록에 추가합니다. 또한 게스트로 비교 목록을 만드는 쇼핑객은 고객으로서 로그인하고 [유지](https://devdocs.magento.com/guides/v2.4/graphql/mutations/assign-compare-list-to-customer.html) 비교 목록

* 가 추가되었습니다. [`generateCustomerTokenAsAdmin`](https://devdocs.magento.com/guides/v2.4/graphql/mutations/generate-customer-token-as-admin.html) 돌연변이 및 업데이트 `Customer` 원격 구매 지원을 지원하기 위한 개체입니다.

* 언어, 장바구니 및 통화 변경과 같은 작업을 지원하기 위해 매장 전체에 현지화 지원이 추가되었습니다. <!-- MC-37801-->

* GraphQL에서 조합에 대한 지원을 추가했습니다. [GitHub-29425](https://github.com/magento/magento2/issues/29425) <!-- ENGCOM-8291 -->

* 많은 변형이 있는 구성 가능한 제품에 대한 제품 데이터 검색을 최적화하기 위해 GraphQL 스키마가 향상되었습니다. <!-- MC-36138-->

* 정수 유형 개체 ID는 대신 `uid` 유형 ID의 속성입니다. <!-- MC-36346-->

* 가 추가되었습니다. `staging` 속성을 [`ProductInterface`](https://devdocs.magento.com/guides/v2.4/graphql/interfaces/product-interface.html) 및 [`CategoryInterface`](https://devdocs.magento.com/guides/v2.4/graphql/interfaces/category-interface.html) 를 클릭하여 제품의 스테이징을 확인하고 관련 캠페인 정보를 확인합니다. <!-- MC-33897-->

자세한 내용은 [GraphQL 개발자 안내서](https://devdocs.magento.com/guides/v2.4/graphql/) 개선 사항에 대한 자세한 내용을 참조하십시오.

* 자산 업로드

* 미디어 갤러리에서 자산 삭제

* 폴더 구조를 관리합니다.

**컨텐츠의 웹에 최적화된 이미지**. 이제 상인들이 사용할 수 있습니다 [웹에 최적화된 이미지 표현물](https://docs.magento.com/user-guide/cms/media-gallery-image-optimization.html) 고해상도 이미지 대신 콘텐츠 형식으로 전송할 수 있습니다. 원본 이미지는 미디어 갤러리에서 수정되지 않은 상태로 유지되며 이미지가 컨텐츠에 삽입될 때 이미지 표현물이 동적으로 생성됩니다.

## 해결된 문제

2.4.2 핵심 코드에서 수백 개의 문제를 해결했습니다.

### 설치, 업그레이드, 배포

<!-- MC-38429-->

* 분할 데이터베이스를 사용하도록 설치를 구성하는 기능은 이 릴리스에서 더 이상 사용되지 않습니다. 자세한 내용은 [Magento Open Source에서 데이터베이스 분할 기능 사용 중단](https://community.magento.com/t5/Magento-DevBlog/Deprecation-of-Split-Database-in-Magento-Commerce/ba-p/465187) 개발 블로그 게시물.

<!-- MC-36472-->

* 이제 판매자는 MySQL 8.x를 실행하는 오픈 소스 배포를 상거래 배포로 성공적으로 업그레이드할 수 있습니다. 이전에는 `AUTO_INCREMENT` 값이 `row_id` 업그레이드 중에 가 추가되었습니다.

<!-- ENGCOM-7280 -->

* 이제 실행 시 오류가 발생할 경우 패치를 생성하는 데 사용된 경로를 식별하는 오류 메시지가 응용 프로그램에 표시됩니다 `bin/magento/setup:db:generate-patch`. [GitHub-27523](https://github.com/magento/magento2/issues/27523)

## 알려진 문제

**문제**: 다음 `[magento_root]/index.php` 파일이 제거되었고 응용 프로그램이 이제 `/pub` 기본적으로 Apache 구성이 제공됩니다. 하위 폴더에서 제공되는 저장소는 예상대로 작동하지 않으며 404 오류를 표시할 수 있습니다. **해결 방법**: symlink를 사용하여 설치를 하위 폴더로 에뮬레이션합니다. 다음 예제에서는 를 사용합니다 `https://shop01.com/shop/` symlink를 사용하여 하위 폴더에 설치를 에뮬레이션하는 방법을 보여 줍니다.

1. 의 하위 디렉토리를 만듭니다. `https://shop01.com/shop/`:

   ```bash
   mkdir magento_root/pub/shop01
   ```

   ```bash
   cd magento_root/pub/shop01
   ```

1. 새로 만든 디렉터리에서 저장소의 상위 디렉터리에 대한 symlink를 만듭니다.

   ```bash
   ln -s ../media media
   ```

   ```bash
   ln -s ../static static
   ```

   ```bash
   ln -s ../../../pub pub
   ```

   ```bash
   ln -s ../.htaccess .htaccess
   ```

   ```bash
   ln -s ../health_check.php health_check.php
   ```

1. 만들기 `index.php` 새 디렉토리 내의 파일(`magento_root/pub/shop01/index.php`)에서 다음 컨텐츠를 추가합니다.

```php
   <?php
   require realpath(__DIR__) . '/../../../app/bootstrap.php';

   switch ($_SERVER['HTTP_HOST']) {
       case 'shop01.com':
       case 'www.shop01.com':
           $params = $_SERVER;
           $params[\Magento\Store\Model\StoreManager::PARAM_RUN_CODE] = 'shop01';
           $params[\Magento\Store\Model\StoreManager::PARAM_RUN_TYPE] = 'website';
           $bootstrap = \Magento\Framework\App\Bootstrap::create(BP, $params);
           $app = $bootstrap->createApplication(\Magento\Framework\App\Http::class);
           $bootstrap->run($app);
           break;

       default:
           $bootstrap = \Magento\Framework\App\Bootstrap::create(BP, $_SERVER);
           /** @var \Magento\Framework\App\Http $app */
           $app = $bootstrap->createApplication(\Magento\Framework\App\Http::class);
           $bootstrap->run($app);
       break;
   }
```

1. 새 하위 디렉토리를 가리키도록 Apache 서버를 구성합니다. 배포 구성은 광범위하게 다를 수 있습니다. 다음은 서버 구성의 예입니다.

```xml
   <VirtualHost *:80>
       DocumentRoot "magento_root/pub/shop1"
       ServerName shop01.com
   </VirtualHost>
```

**문제**: 쇼핑객이 처음 지급 페이지의 CAPTCHA 필드에 올바른 코드를 입력했을 때 애플리케이션은 예상대로 결제 정보를 제출하지 않지만 다음 오류를 표시합니다.  `There has been an error processing your request`. 이 문제는 쇼핑객이 CAPTCHA 코드를 입력하지 않고 또는 잘못 입력하여 순서를 지정하려고 할 때만 발생합니다.  **해결 방법**: 페이지를 새로 고칩니다. <!-- MC-40506-->

## 마이그레이션 도구 키트

데이터 마이그레이션 도구는 기존 Magento 1.x 저장소 데이터를 Magento 2.x로 전송하는 데 도움이 됩니다. 이 명령줄 인터페이스에는 확인, 진행 추적, 로깅 및 테스트 기능이 포함됩니다. 설치 지침은 [데이터 마이그레이션 도구 설치](../../../tools/data-migration-tool/install.md). 탐색 또는 참가 고려 [데이터 마이그레이션 저장소](https://github.com/magento/data-migration-tool).

다음 [코드 마이그레이션 툴킷](https://github.com/magento-commerce/code-migration) 기존 Magento 1.x 저장소 확장 및 사용자 지정을 Magento 2.x에 전송하는 데 도움이 됩니다. 명령줄 인터페이스는 Magento 1.x 모듈 및 레이아웃을 변환하는 스크립트를 포함합니다.
