# Adobe Commerce 2.4.7-beta2 릴리스 노트

>[!IMPORTANT]
>
>베타 릴리스에는 결함이 포함될 수 있으며 어떠한 종류의 보증도 없이 &quot;있는 그대로&quot; 제공됩니다. Adobe은 베타 릴리스를 유지, 수정, 업데이트, 변경, 수정 또는 지원(Adobe 지원 서비스 등을 통해)할 의무가 없습니다. 고객은 베타 릴리스 및/또는 관련 설명서나 자료의 올바른 기능이나 성능에 어떠한 의존도 하지 말고 주의하는 것이 좋습니다. 따라서 Beta 릴리스를 사용하는 것은 전적으로 고객 자신의 책임입니다.

Adobe Commerce 2.4.7-beta2에는 200개 이상의 품질 수정 사항 및 개선 사항이 포함되어 있습니다. Core Composer 종속성 및 타사 라이브러리가 사용 가능한 최신 버전으로 업그레이드되었습니다.

이번 릴리스로 인해 사용자 지정 특성 및 GraphQL resolver 캐시에 대한 GraphQL 적용 범위가 늘어났습니다. 최근 업데이트된 FedEx 및 UPS 서비스에 대한 지원도 도입한다.

### 향상된 성능 및 확장성

이 릴리스에서는 대규모 배포 구성을 위한 저장 작업의 처리 속도가 크게 향상되었습니다.

**2.4.7-Beta2에 대한 새로운 개선 사항**

* **이제 나머지 항목에 대해 JSON 형식이 지원됩니다 [API 가져오기](https://developer.adobe.com/commerce/webapi/rest/modules/import/)**. 판매자는 이제 분당 최대 100,000개의 레코드를 JSON 형식으로 Adobe Commerce에 가져올 수 있습니다. <!-- ACPT-943 -->

* **영업 규칙 성능 개선 사항**. 많은(약 100,000개) 활성 영업 규칙을 사용하여 엔터프라이즈 배포 성능을 개선했습니다.  프로모션을 많이 구현하는 엔터프라이즈 배포는 많은 활성 장바구니 규칙을 배포하는 경우가 많습니다. Commerce 2.4.7을 실행하는 이러한 유형의 엔터프라이즈 배포에서는 체크아웃 작업 중 구성된 장바구니 가격 규칙 수와 관련된 성능 저하가 발생하지 않습니다. <!-- ACPT-1198 -->

**2.4.7-beta1에 도입된 성능 개선 사항**

Adobe Commerce 2.4.7의 이전 베타 버전:

* **여러 스토어가 있는 배포를 위한 스토어 수준 구성의 보다 빠른 저장 작업**. 저장소가 500개를 초과하는 배포에 구성 설정을 저장하는 것은 시간이 많이 걸릴 수 있습니다. 새로운 [비동기 구성](https://experienceleague.adobe.com/docs/commerce-operations/performance-best-practices/configuration.html?lang=en#asynchronous-configuration-save-2.4.7-beta1) 모듈은 소비자를 사용하여 메시지 큐에서 저장 작업을 처리하는 cron 작업을 실행하여 비동기 구성 저장 작업을 활성화합니다. AsyncConfig는 기본적으로 비활성화되어 있습니다.

* **대규모 구성을 위한 구성 캐시 생성 속도 향상**. 다음 `bin/magento cache:clean config` 이제 명령은 구성 캐시가 활성화되면 구성 캐시를 미리 warm합니다. 따라서 대규모 구성에 대한 구성 캐시를 생성하는 데 필요한 가동 중지 시간이 줄어듭니다. 구성 저장 작업이 더 이상 `config_scopes` 캐시에 데이터를 쓰기 전에 캐시합니다. 이렇게 하면 구성 데이터를 쓰는 동안 다른 요청이 잠기는 시간도 줄어듭니다.

### GraphQL API용 Application Server

Commerce 2.4.7-beta2에서는 Openswool PHP 프레임워크에 구현된 GraphQL API용 새 PHP Application Server를 도입했습니다. 이 애플리케이션 서버를 사용하면 Adobe Commerce이 Commerce GraphQL API 요청 중 상태를 유지하고 부트스트래핑을 사용하지 않으므로 모든 GraphQL API에 대해 더 높은 처리량, 더 낮은 지연, 효율적인 리소스 사용이 가능합니다. 프로세스 간에 애플리케이션 상태를 공유하면 GraphQL API 요청의 효율성이 훨씬 향상되고 GraphQL API 응답이 평균 30% 더 빨라집니다.

다음 `ApplicationServer` 모듈은 PHP 응용 프로그램 서버를 구현합니다. 온-프레미스 배포에서 이 모듈을 구현하려는 개발자도 Swool 및 Nginx를 설치 및 구성해야 합니다. 클라우드 인프라 프로젝트에서 Adobe Commerce을 사용하는 개발자는 이미 Nginx 및 OpenSwool이 인프라에 포함되어 있으므로 이러한 구성 요소를 다운로드할 필요가 없습니다. 클라우드 개발자는 애플리케이션 서버를 수동으로 활성화해야 하며 다른 서비스와 같이 모듈을 시작하거나 중지할 수 없습니다. 모듈을 시작하거나 중지하려면 Adobe Commerce을 재배포해야 하며, 이를 위해서는 사이트 가동 중지 시간이 필요합니다.

다음을 참조하십시오 [GraphQL API용 Application Server](https://experienceleague.adobe.com/docs/commerce-operations/performance-best-practices/performance-best-practices/application-server.html) 온프레미스 및 클라우드 배포 모두에 해당합니다.

GraphQL API 모듈용 PHP Application Server의 베타 버전입니다. 사용자 지정 및 확장을 포함하는 배포는 모듈과 호환되지 않을 수 있습니다. 모듈을 사용하는 판매자는 배포된 사용자 지정 및 확장과 호환되지 않을 수 있는 문제를 해결할 준비를 해야 합니다. <!-- ACPT-27 -->

* 이제 Adobe Commerce에서 다음을 성공적으로 반환합니다. `content` 의 특성(base64 이미지 코드) `GET V1/products/sku/media` product media API 응답. 다음 `content` 속성은 이전에 응답에 나타나지 않았습니다.

### 위시리스트

<!-- ACP2E-1784 -->

* 이제 구매자가 동일한 항목을 위시리스트에 여러 번 추가하려고 할 때 Commerce에 항목이 이미 위시리스트에 추가되었음을 나타내는 메시지가 표시됩니다. 이전에는 이러한 조건에서 Commerce가 위시리스트의 제품 수량을 증가했습니다.

<!-- AC-8610 -->

* 이제 Adobe Commerce은 업로드해야 하는 사용자 정의 가능 옵션 파일이 있는 제품을 위시리스트에 성공적으로 추가했습니다. 이전에는 Commerce가 위시리스트에 하나가 아닌 두 개의 제품을 추가했습니다. [GitHub-37437](https://github.com/magento/magento2/issues/37437)
