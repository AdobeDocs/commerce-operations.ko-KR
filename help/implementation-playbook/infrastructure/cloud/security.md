# 보안

Adobe Commerce Pro 계획 아키텍처는 매우 안전한 환경을 제공하도록 설계되었습니다. 각 고객은 다른 고객과 구분하여 자체 분리된 서버 환경에 배포됩니다. 프로덕션 환경의 보안 세부 사항은 아래에 설명되어 있습니다.

## 웹 브라우저

클라우드 환경에서 들어오고 나가는 대부분의 트래픽은 고객의 웹 브라우저에서 가져옵니다. 웹 사이트의 모든 페이지에 대해 HTTPS를 사용하여 소비자 트래픽을 보호할 수 있습니다(공유 SSL 인증 또는 추가 요금으로 고객의 자체 SSL 인증서 사용). 체크아웃 및 계정 페이지는 항상 HTTPS를 사용하여 제공됩니다. 가장 좋은 방법은 HTTPS에서 모든 페이지를 제공하는 것입니다.

## CDN(콘텐츠 전달 네트워크)

기본적으로 CDN 및 DDoS(Distributed Denial of Service) 보호를 제공합니다. 기본 CDN은 원본 서버에 대한 직접 액세스를 격리하는 데 도움이 됩니다. 공용 DNS는 기본 네트워크만 가리킵니다. 강력한 DDoS 솔루션은 더욱 복잡한 Layer 7 공격뿐만 아니라 Layer 3 및 Layer 4 공격으로부터 보호합니다. 계층 7 공격은 전체 HTTP/HTTPS 요청을 기반으로 하며 헤더, 쿠키, 요청 경로, 클라이언트 IP 또는 지리적 위치와 같은 표시기를 포함한 클라이언트 및 요청 기준에 따라 사용자 지정 규칙을 사용하여 차단될 수 있습니다.

## 웹 응용 프로그램 방화벽(WAF)

WAF(Apple Web Application Firewall)를 사용하여 추가 보호를 제공합니다. Apple의 클라우드 기반 WAF는 OWASP 코어 규칙 세트와 같은 상업적 및 오픈 소스 소스의 타사 규칙을 사용합니다. 또한 Adobe Commerce 관련 규칙이 적용됩니다. 고객은 주입 공격과 악성 입력, 교차 사이트 스크립팅, 데이터 추출, HTTP 프로토콜 위반 및 기타 OWASP Top 10 위협 등 주요 애플리케이션 계층 공격으로부터 보호됩니다.

WAF 규칙은 소프트웨어 패치 전에 Managed Services에서 &quot;사실상&quot; 보안 문제를 해결할 수 있는 새로운 취약점이 발견되면 Adobe Commerce에 의해 업데이트됩니다. 실제 WAF에서는 속도 제한 또는 보트 탐지 서비스를 제공하지 않습니다. 원하는 경우 고객은 Apple과 호환되는 타사 보트 탐지 서비스에 라이센스를 부여할 수 있습니다.

## Virtual Private Cloud(VPC)

Adobe Commerce Pro 계획 프로덕션 환경은 VPC(Virtual Private Cloud)로 구성되므로 프로덕션 서버는 격리되고 클라우드 환경 내외를 연결할 수 있는 기능이 제한됩니다. 클라우드 서버에 대한 보안 연결만 허용됩니다. SFTP 또는 rsync와 같은 보안 프로토콜은 파일 전송에 사용할 수 있습니다.

고객은 SSH 터널을 사용하여 애플리케이션과의 통신을 보호할 수 있습니다. AWS PrivateLink는 추가 요금으로 이용 가능합니다. 이러한 서버에 대한 모든 연결은 환경에 대한 연결을 제한하는 가상 방화벽인 AWS 보안 그룹을 사용하여 제어됩니다. 고객의 기술 리소스는 SSH를 사용하여 이러한 서버에 액세스할 수 있습니다.

## 암호화

Amazon EBS(Elastic Block Store)는 스토리지에 사용됩니다. 모든 EBS 볼륨은 AES-265 알고리즘을 사용하여 암호화됩니다. 즉, 데이터는 안전하게 암호화됩니다. 또한, 시스템은 CDN과 원본 간, 원본 서버 간에 데이터를 전송 중일 때 암호화합니다. 고객 암호는 해시로 저장됩니다. 결제 게이트웨이에 대한 자격 증명을 포함한 중요한 자격 증명은 SHA-256 알고리즘을 사용하여 암호화됩니다.

Adobe Commerce 응용 프로그램은 데이터가 서버에 없거나 서버 간에 전송되지 않는 경우 열 또는 행 수준 암호화 또는 암호화를 지원하지 않습니다. 고객은 애플리케이션 내에서 암호화 키를 관리할 수 있습니다. 시스템에서 사용하는 키는 AWS 키 관리 시스템에 저장되며 서비스의 일부를 제공하려면 Managed Services에서 관리해야 합니다.

## 침투 테스트

Managed Services은 기본 제공 애플리케이션을 사용하여 Adobe Commerce 클라우드 시스템의 정기적인 침투 테스트를 수행합니다. 고객은 맞춤형 애플리케이션의 모든 침투 테스트를 담당합니다.

## 결제 게이트웨이

Adobe Commerce을 사용하려면 신용 카드 데이터가 소비자의 브라우저에서 결제 게이트웨이로 직접 전달되는 결제 게이트웨이 통합이 필요합니다. Adobe Commerce Pro 계획 Managed Services 환경에서 카드 데이터를 사용할 수 없습니다. 전자 상거래 응용 프로그램의 트랜잭션에 대한 작업은 게이트웨이의 트랜잭션 참조를 사용하여 완료됩니다.

## Adobe Commerce 애플리케이션

Adobe은 보안 취약점에 대한 코어 애플리케이션 코드를 정기적으로 테스트합니다. 결함 및 보안 문제에 대한 패치는 고객에게 제공됩니다. 제품 보안 팀은 OWASP 응용 프로그램 보안 지침에 따라 Adobe Commerce 제품의 유효성을 검사합니다. 몇 가지 보안 취약성 평가 도구와 외부 공급업체는 규정 준수를 테스트하고 확인하는 데 사용됩니다. 보안 도구는 다음과 같습니다.

- Veracode 정적 및 동적 스캐닝
- RIPS 소스 코드 스캔
- Trustwave 및 Alert Logic의 취약성 스캔 서비스
- 버 스위트 프로
- 오아스프자프
- 및SqlMap

전체 코드 베이스는 2주에 걸쳐 이러한 도구를 사용하여 검사됩니다. 고객은 직접 전자 메일, 애플리케이션의 알림 및 [보안 센터](https://magento.com/security).

고객은 PCI 지침에 따라 릴리즈 후 30일 이내에 맞춤형 애플리케이션에 이러한 패치를 적용해야 합니다. Adobe은 [보안 검사 도구](https://docs.magento.com/user-guide/magento/security-scan.html) 이 기능을 통해 가맹점은 정기적으로 사이트를 모니터링하고 알려진 보안 위험, 맬웨어 및 무단 액세스에 대한 업데이트를 받을 수 있습니다. 보안 스캔 도구는 무료 서비스이며 모든 버전의 Adobe Commerce에서 실행할 수 있습니다.

보안 연구원이 취약점을 식별하고 보고하도록 유도하기 위해 Adobe Commerce은 [버그 보상 프로그램](https://hackerone.com/magento) 내부 테스트 외에 추가로 사용할 수 있습니다. 또한 고객이 원하는 경우 고객이 직접 검토할 수 있도록 애플리케이션의 전체 소스 코드를 제공합니다.

## 읽기 전용 파일 시스템

모든 실행 코드가 읽기 전용 파일 시스템 이미지에 배포되어 공격을 위해 사용할 수 있는 표면을 크게 줄입니다. 배포 프로세스는 스쿼시-FS 이미지를 생성합니다. 이 접근 방식을 사용하면 PHP 또는 JavaScript 코드를 시스템에 주입하거나 Adobe Commerce 애플리케이션 파일을 수정할 수 있는 기회를 크게 줄일 수 있습니다.

## 원격 배포

Managed Services 프로덕션 환경으로 실행 코드를 가져올 수 있는 유일한 방법은 프로비저닝 프로세스를 통해 실행하는 것입니다. 소스 저장소에서 원격 저장소로 소스 코드를 푸시하여 배포 프로세스를 시작합니다. 해당 배포 대상에 대한 액세스가 제어되므로 배포 대상에 액세스할 수 있는 사용자를 완전히 제어할 수 있습니다. 비프로덕션 환경에 대한 모든 애플리케이션 코드 배포는 고객이 제어합니다.

## 로깅

모든 AWS 활동이 AWS CloudTrail에 기록됩니다. Linux, 애플리케이션 서버 및 데이터베이스 로그는 운영 서버에 저장되고 백업에 저장됩니다. 모든 소스 코드 변경 사항은 Git 리포지토리에 기록됩니다. 배포 내역은 Adobe Commerce에서 사용할 수 있습니다 [Project Web Interface](https://devdocs.magento.com/cloud/project/projects.html#login). 모든 지원 액세스가 기록되고 지원 세션이 기록됩니다.

## 중요 데이터

중요한 데이터는 소비자의 개인 정보 또는 Managed Services 고객의 기밀 데이터를 포함할 수 있습니다. 중요한 고객 및 소비자 데이터 보호는 Adobe Commerce Managed Services에 대한 중요한 의무입니다. Managed Services과 Adobe 고객 모두 개인 식별 정보에 대한 법적 책임을 집니다. 아키텍처의 보안 기능 외에도 중요한 데이터에 대한 배포 및 액세스를 제한하는 다른 제어 기능이 있습니다.

고객은 자신의 데이터를 소유하고 해당 데이터가 위치할 위치를 제어할 수 있습니다. 고객은 프로덕션 및 개발 인스턴스가 상주하는 위치를 지정합니다. 또한 Commerce와 함께 Adobe Commerce 보고 환경에 사용할 위치와 Adobe Commerce 보고 애플리케이션이 개인 식별 정보에 액세스할 수 있는지 여부를 지정합니다. 프로덕션 인스턴스는 대부분의 AWS 지역에 있지만, 개발 및 Adobe Commerce 보고 환경은 미국 또는 유럽 연합에서 찾을 수 있습니다.

중요한 데이터는 Apple CDN 서버 네트워크를 통해 전달될 수 있지만 실제 네트워크에 저장되지는 않습니다. Adobe Commerce Managed Services 오퍼링에 포함된 모든 파트너는 중요한 데이터를 보호하기 위한 계약 의무를 갖습니다. Managed Services은 고객이 지정한 위치에서 중요한 고객 또는 소비자 데이터를 이동하지 않습니다.

Adobe Commerce Managed Services 서비스에 포함된 서비스를 제공하는 과정의 일부로, Managed Services 직원은 중요한 데이터가 포함된 프로덕션 시스템에 액세스해야 합니다. 이러한 직원들은 중요한 고객 및 소비자 데이터를 보호하기 위한 의무를 숙지합니다. 이러한 시스템에 대한 액세스는 액세스가 필요한 직원만 사용할 수 있습니다. 이러한 직원들은 이러한 서비스를 제공하는 데 필요한 시간에만 액세스할 수 있습니다.

## 일반 데이터 보호 규정(GDPR)

GDPR은 유럽 연합(EU)에서 개인 정보를 수집하고 처리하는 데 대한 지침을 설정하는 법률 프레임워크입니다. 이러한 규정은 사이트가 어디에서 구축되었는지에 관계없이 적용되며, EU 방문자가 사이트에 액세스할 수 있습니다.

기본적으로 방문자는 사이트에서 수집하는 데이터에 대한 알림을 받아야 하며 정보 수집에 명시적으로 동의해야 합니다. 사이트에서 보유하고 있는 개인 데이터가 위반되면 사이트는 방문자에게 알려야 합니다.

사이트를 운영하는 머천트 또는 회사에는 사이트의 데이터 보안을 담당하는 전용 데이터 보호 담당자도 있어야 하며, 방문자가 데이터를 삭제하도록 요청할 경우 이 개인(또는 웹 사이트 관리 팀)은 연락처로 사용할 수 있어야 합니다.

또한 GDPR은 수집된 개인 식별 정보(예: 이름, 인종 및 생년월일)를 익명 처리하거나 익명을 조건으로 사용해야 합니다.

>[!NOTE]
>
> 이 페이지는 GDPR에서 고려해야 할 사항에 대한 일반적인 개요로 사용됩니다. 자세한 내용은 법률 자문을 구하거나[공식 텍스트](https://eur-lex.europa.eu/eli/reg/2016/679/oj).

## 백업

최근 24시간 동안 매시간마다 백업이 수행됩니다. 24시간 후에 AWS EBS 스냅샷 서비스를 사용하여 다음 일정에 따라 백업이 유지됩니다.

| 기간 | 백업 보존 정책 |
|----------------|-------------------------|
| 일 1~3 | 각 백업 |
| 4일~6일 | 하루에 백업 1회 |
| 2-6주 | 매주 1회 백업 |
| 8주~12주 | 매주 2회 백업 |
| 12~22주 | 매월 1회 백업 |

이렇게 하면 중복 스토리지에 독립적인 백업이 만들어집니다. EBS 볼륨이 암호화되므로 백업도 암호화됩니다. 또한 Managed Services에서는 고객이 요청한 온디맨드 백업을 수행합니다.

Adobe Commerce Managed Services 백업 및 복구 방식은 전체 시스템 백업과 결합된 고가용성 아키텍처를 사용합니다. 각 프로젝트는 세 개의 개별 AWS 가용 영역에 걸쳐 모든 데이터, 코드 및 자산을 복제합니다. 별도의 데이터 센터가 있는 각 영역
